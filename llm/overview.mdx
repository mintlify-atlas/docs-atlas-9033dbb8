Typia's LLM integration module provides a complete harness for building reliable AI-powered applications. It generates function calling schemas directly from TypeScript classes, validates every argument the LLM produces, and offers utilities to recover from malformed JSON and guide the model toward correct outputs through structured feedback. This page covers what the module does, the problem it solves, and guides you to the right page for each feature.

## The problem Typia solves

Large language models are unreliable JSON producers. When asked to call a function, a raw LLM response often contains:

- **Malformed JSON** — trailing commas, JavaScript-style comments, unquoted keys, or truncated strings.
- **Type errors** — numbers returned as strings (`"42"` instead of `42`), missing required properties, or hallucinated extra fields.
- **Wrong function selection** — without precise schemas, the LLM picks the wrong tool or invents arguments.

Without a validation layer, these issues surface as runtime crashes deep inside your application logic. Typia intercepts them at the boundary, giving the LLM a chance to self-correct before any function executes.

<Note>
  Typia improves LLM function calling accuracy from **6.75% → 100%** through built-in argument validation and structured error feedback.
</Note>

## Supported model families

Typia's LLM schemas are compatible with all major model providers:

| Provider | Model type identifier |
|---|---|
| OpenAI (ChatGPT, GPT-4o) | `"chatgpt"` |
| Anthropic (Claude) | `"claude"` |
| Google (Gemini) | `"gemini"` |
| Meta (Llama) | `"llama"` |
| Generic JSON Schema 3.0 | `"3.0"` |
| Generic JSON Schema 3.1 | `"3.1"` |

The model type is passed as the `Config` generic argument or inferred from the config object you provide at call sites.

## API surface at a glance

### Function calling

| Function | What it returns | When to use |
|---|---|---|
| `typia.llm.application<Class>()` | `ILlmApplication<Class>` — schemas only | Build schemas once, execute manually |
| `typia.llm.controller<Class>(name, exec)` | `ILlmController<Class>` — schemas + executor | Bind a class instance for direct execution |
| `typia.llm.parameters<P>()` | `ILlmSchema.IParameters` — single params schema | Low-level schema for one function |
| `typia.llm.schema<T>()` | `ILlmSchema` — single type schema | Embed a type schema in a larger structure |

### Structured output

| Function | What it returns | When to use |
|---|---|---|
| `typia.llm.structuredOutput<T>()` | `ILlmStructuredOutput<T>` | Constrain LLM to produce a single typed object |

### JSON utilities

| Function | What it does |
|---|---|
| `typia.llm.parse<P>(str)` | Lenient JSON parse + type coercion |
| `typia.llm.coerce<P>(obj)` | Type coercion on already-parsed objects |
| `typia.llm.createParse<P>()` | Returns a reusable parser (avoids repeated schema generation) |
| `typia.llm.createCoerce<P>()` | Returns a reusable coercer |

### Error feedback

`LlmJson.stringify(failure)` from `@typia/utils` formats a `IValidation.IFailure` as annotated JSON with inline `// ❌` comments, ready to send back to the LLM for self-correction.

## How validation improves accuracy

Every `ILlmFunction` generated by `typia.llm.application()` carries a built-in `validate` function. The typical correction loop looks like this:

<Steps>
  <Step title="Generate and register schemas">
    Call `typia.llm.application<MyService>()` once at startup. Pass `application.functions` to the LLM provider as the tool list.
  </Step>
  <Step title="Receive LLM tool call">
    The LLM selects a function and returns raw JSON arguments.
  </Step>
  <Step title="Parse and coerce">
    Call `fn.parse(rawJson)` to handle malformed JSON and coerce string-typed numbers/booleans to their proper types.
  </Step>
  <Step title="Validate">
    Call `fn.validate(args)`. If validation succeeds, execute the function. If it fails, continue to the next step.
  </Step>
  <Step title="Feed errors back to the LLM">
    Call `LlmJson.stringify(failure)` and include the annotated JSON in the next message. The LLM sees exactly which properties are wrong and why, then produces corrected arguments.
  </Step>
</Steps>

Typia's API surface is small, but understanding the model behind it makes every feature predictable. This page covers how TypeScript types become runtime code, what type tags are and how they add constraints, how `is`, `assert`, and `validate` differ, and when to use the `createX` factory functions.

## The transformation model

Every Typia function is a stub at the source level. The actual implementation is generated by the TypeScript transformer plugin at compile time.

When `tsc` (patched by `ts-patch`) or a bundler plugin processes a file containing a Typia call, the transformer reads the generic type argument, walks the TypeScript type tree, and replaces the function call with specialized inline JavaScript. The original `import typia from "typia"` call is rewritten or removed in the output.

Consider this source file:

```typescript
// Source
import typia from "typia";

export const checkString = typia.createIs<string>();
```

After compilation the transformer emits:

```javascript
// Output
import typia from "typia";

export const checkString = (() => {
  return (input) => "string" === typeof input;
})();
```

The type `string` was analyzed at compile time and replaced with an equivalent runtime expression. For a complex nested type, the output would be a correspondingly detailed series of property checks â€” still plain JavaScript, with no schema objects or runtime reflection.

This is why Typia is so fast: the generated code is equivalent to what you would write by hand for each specific type. There is no interpreter reading a schema at runtime.

<Note>
  The transformer must be active during compilation for any Typia function to work. A call to `typia.is<T>()` in a file that was not processed by the transformer will throw a `NoTransformConfigurationError` at runtime. See [Setup](/setup) to configure the transformer.
</Note>

## Pure TypeScript types as the source of truth

Typia works with plain TypeScript interfaces and type aliases. You do not need classes, decorators, or constructor functions.

```typescript
import typia, { tags } from "typia";

// Plain interface â€” no class, no decorators
interface IOrder {
  id: string & tags.Format<"uuid">;
  productId: string & tags.Format<"uuid">;
  quantity: number & tags.Minimum<1> & tags.Type<"uint32">;
  note?: string & tags.MaxLength<500>;
}

const order = typia.assert<IOrder>(rawInput);
```

Union types, intersection types, tuple types, recursive types, and `readonly` modifiers are all supported. The transformer resolves the full type structure, including types imported from other modules.

```typescript
type Nullable<T> = T | null;
type ID = string & tags.Format<"uuid">;

interface IUser {
  id: ID;
  name: string;
  supervisor: Nullable<IUser>; // recursive type
}

typia.assert<IUser>(input); // works correctly
```

## Type tags: validation constraints on types

Type tags are TypeScript intersection types that carry constraint metadata recognized by Typia's transformer. They are invisible to normal TypeScript code â€” they have no runtime representation â€” but Typia reads them at compile time and generates the corresponding validation logic.

All built-in tags are exported from `typia`'s `tags` namespace:

```typescript
import { tags } from "typia";
```

### Numeric constraints

```typescript
interface IProduct {
  price: number & tags.Minimum<0>;            // price >= 0
  discount: number & tags.ExclusiveMaximum<1>; // discount < 1
  quantity: number & tags.Type<"uint32">;      // unsigned 32-bit integer
  weight: number & tags.MultipleOf<0.5>;       // must be a multiple of 0.5
}
```

### String constraints

```typescript
interface IProfile {
  email: string & tags.Format<"email">;          // valid email address
  website: string & tags.Format<"url">;          // valid URL
  userId: string & tags.Format<"uuid">;          // UUID v1â€“v5
  bio: string & tags.MinLength<10> & tags.MaxLength<500>;
  slug: string & tags.Pattern<"^[a-z0-9-]+$">; // lowercase slug
}
```

### Array constraints

```typescript
interface ICart {
  items: Array<string> & tags.MinItems<1> & tags.MaxItems<50>;
  tags: Array<string> & tags.UniqueItems;
}
```

Tags are standard TypeScript types and are fully compatible with type inference, generics, and `typeof`. There is nothing Typia-specific about how they appear in your types at the TypeScript level.

## `is`, `assert`, and `validate` â€” choosing the right function

Typia provides three validator variants with different error-handling behaviors. Choose based on what you need to do when validation fails.

### `typia.is<T>(input)` â€” boolean type guard

`is` returns `true` if the input conforms to `T`, and `false` otherwise. It acts as a TypeScript type predicate, narrowing the type in `if` branches. It stops at the first mismatch it finds.

```typescript
import typia from "typia";

const input: unknown = JSON.parse(body);

if (typia.is<IOrder>(input)) {
  // TypeScript knows input is IOrder here
  processOrder(input);
} else {
  return res.status(400).send("Invalid order");
}
```

Use `is` when you only need a pass/fail result and you do not need to know _what_ failed.

### `typia.assert<T>(input)` â€” throws on first error

`assert` returns the input typed as `T` if valid. On the first type mismatch it throws a `TypeGuardError` that contains:

- `path` â€” property path of the failure (e.g., `"input.items[2].quantity"`)
- `expected` â€” expected type description (e.g., `"number & Minimum<1>"`)
- `value` â€” the actual value that failed

```typescript
import typia from "typia";

try {
  const order = typia.assert<IOrder>(rawInput);
  // order is typed as IOrder
  processOrder(order);
} catch (e) {
  if (e instanceof typia.TypeGuardError) {
    console.error(`Validation failed at ${e.path}: expected ${e.expected}`);
  }
}
```

You can provide a custom error factory to throw a different error type:

```typescript
const order = typia.assert<IOrder>(rawInput, (props) =>
  new BadRequestError(`Field ${props.path} is invalid`)
);
```

Use `assert` when you want to fail fast and report the first error â€” the common case for API request validation.

### `typia.validate<T>(input)` â€” collects all errors

`validate` never throws. It returns an `IValidation<T>` object:

- `{ success: true, data: T }` when the input is valid
- `{ success: false, errors: IValidation.IError[] }` when one or more checks fail

Each error in the array has the same `path`, `expected`, and `value` fields as `TypeGuardError`.

```typescript
import typia from "typia";

const result = typia.validate<IOrder>(rawInput);

if (result.success) {
  processOrder(result.data);
} else {
  const messages = result.errors.map(
    (e) => `${e.path}: expected ${e.expected}, got ${JSON.stringify(e.value)}`
  );
  return res.status(400).json({ errors: messages });
}
```

Use `validate` when you want to show all validation errors at once â€” useful for form validation or detailed API error responses.

### Quick comparison

| Function | Return type | Throws? | Stops at first error? |
|---|---|---|---|
| `typia.is<T>` | `boolean` (type predicate) | No | Yes |
| `typia.assert<T>` | `T` | Yes (`TypeGuardError`) | Yes |
| `typia.validate<T>` | `IValidation<T>` | No | No (collects all) |

There are also strict equality variants (`equals`, `assertEquals`, `validateEquals`) that additionally reject properties not defined in `T`.

## `createX` factory functions

Every validator has a corresponding `createX` factory function: `createIs`, `createAssert`, `createValidate`, `createAssertEquals`, and so on.

A direct call such as `typia.is<User>(input)` is transformed at every call site. A `createX` call is transformed once, producing a reusable function that can be called many times with no re-compilation overhead.

```typescript
import typia from "typia";

interface IUser {
  id: string;
  name: string;
}

// Transformed once at module load
const isUser = typia.createIs<IUser>();
const assertUser = typia.createAssert<IUser>();
const validateUser = typia.createValidate<IUser>();

// Called many times with no overhead
for (const item of items) {
  if (isUser(item)) {
    // ...
  }
}
```

`createValidate` also implements the [Standard Schema](https://standardschema.dev/) interface (`StandardSchemaV1`), making it compatible with frameworks and libraries that accept Standard Schema validators.

<Tip>
  Prefer `createX` functions when you validate the same type repeatedly â€” for example, in middleware, in a service method called per request, or in a tight loop. The performance difference is negligible for occasional validations but measurable at scale.
</Tip>

## Mental model for using Typia

Think of Typia in three layers:

1. **Types** â€” pure TypeScript interfaces and type aliases, optionally enriched with type tags for constraints. These are your only definitions. You do not write schemas, decorators, or codec objects.

2. **Transformer** â€” a TypeScript compiler plugin that reads your types and generates runtime code. It runs once during compilation. Its output is plain JavaScript.

3. **Functions** â€” the `is`, `assert`, `validate`, `createIs`, etc. exported from `typia`. These are stubs that the transformer replaces. At runtime, they are the generated code.

When something does not validate as expected, the right place to look is your TypeScript type definition â€” and if needed, the type tag constraints attached to it. The generated code faithfully reflects whatever the type says.
